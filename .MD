## 1. What This App Does

**Vendors Dashboard** is a private admin tool for **Merch By DZ**, an Algerian print-on-demand platform.

The platform works like this:
- **Vendors** are affiliate partners who market and sell products
- **Merch By DZ** handles production and shipping
- When a client places an order on WooCommerce, it flows automatically into this dashboard via webhook
- An employee then fills in the cost prices
- The system calculates profits for both Merch By DZ and the vendor
- Payouts are tracked and sent to vendors

## 3. Pricing Model (CRITICAL вЂ” Read Carefully)

There are **5 price fields** per order. Understanding these is the core of the entire system.

### Fields

| Field name | Label | Description |
|---|---|---|
| `prix_client` | Prix Client | Final price the client paid. Comes from WooCommerce automatically via webhook. |
| `prix_fournisseur` | Prix Fournisseur | What Merch By DZ pays the supplier. Entered manually by employee after order arrives. |
| `benefice_merch` | BГ©nГ©fice Merch | Merch By DZ's profit margin. Entered manually by employee after order arrives. |
| `prix_vendeur` | Prix Vendeur | What the vendor sees as the "product price". **Computed automatically.** |
| `benefice_vendeur` | BГ©nГ©fice Vendeur | The vendor's profit. **Computed automatically.** |

### Formulas (computed columns in PostgreSQL)

```
prix_vendeur    = prix_fournisseur + benefice_merch
benefice_vendeur = prix_client - prix_vendeur
```

### Who sees what

| Role | Sees |
|---|---|
| Admin / Employee | Everything вЂ” all 5 fields |
| Vendor | Only `prix_vendeur` вЂ” they never see `prix_fournisseur` or `benefice_merch` |

### Example

```
prix_client       = 3500 DA   в†ђ comes from WooCommerce
prix_fournisseur  =  800 DA   в†ђ entered by employee
benefice_merch    =  200 DA   в†ђ entered by employee
в”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђ
prix_vendeur      = 1000 DA   в†ђ computed (800 + 200)
benefice_vendeur  = 2500 DA   в†ђ computed (3500 - 1000)
```

### Workflow per order

1. WooCommerce order is placed by a client
2. Webhook fires в†’ order inserted into DB with `prix_client` and `status = EN_LIVRAISON`
3. `prix_fournisseur` and `benefice_merch` default to `0`
4. Employee opens the order in the dashboard
5. Employee enters `prix_fournisseur` and `benefice_merch`
6. DB auto-computes `prix_vendeur` and `benefice_vendeur` instantly
7. Dashboard flags orders where prices are still `0` (needs attention)

### SQL Views

#### `vendor_stats`
Aggregated performance per vendor.

```sql
SELECT
  vendor_id,
  vendor_name,
  total_orders,
  delivered_orders,       -- COUNT WHERE status = LIVREE
  retour_orders,          -- COUNT WHERE status = RETOUR
  total_benefice,         -- SUM(benefice_vendeur) WHERE LIVREE
  total_retour_loss,      -- SUM(prix_vendeur) WHERE RETOUR
  net_vendor_profit,      -- total_benefice - total_retour_loss
  merch_total_profit,     -- SUM(benefice_merch) WHERE LIVREE
  total_revenue           -- SUM(prix_client) WHERE LIVREE
FROM vendors LEFT JOIN orders ...
```

#### `vendor_unpaid_orders`
Orders eligible to be included in a payout.

Conditions:
- `is_vendor_paid = false`
- `status IN ('LIVREE', 'RETOUR')`
- Not already linked to any payout via `payout_orders`

#### `dashboard_kpis`
Single-row global stats for the overview page.

---

### Payout Calculation Logic

```
net_payout = SUM(benefice_vendeur WHERE status = 'LIVREE')
           - SUM(prix_vendeur WHERE status = 'RETOUR')
```

**Why prix_vendeur for retours?**
When an order is returned, Merch By DZ already paid the supplier. The loss is `prix_vendeur` (the full production cost visible to vendor), which gets deducted from the vendor's payout.

## 5. Webhook (WooCommerce в†’ Supabase)

### Setup in WooCommerce

```
WooCommerce в†’ Settings в†’ Advanced в†’ Webhooks в†’ Add Webhook
Topic:        Order created
              Order updated (add second webhook)
Delivery URL: https://your-domain.vercel.app/api/webhook
Secret:       [same as WEBHOOK_SECRET env var]
```

### What the webhook receives

The webhook handler reads from WooCommerce `meta_data` array:

| Meta key | Maps to DB field | Notes |
|---|---|---|
| `vendor_id` | `vendor_id` | UUID of the vendor |
| `vendor_name` | `vendor_name` | Used for auto-creation if vendor doesn't exist |
| `payload.total` | `prix_client` | WooCommerce order total вЂ” what client paid |

`prix_fournisseur` and `benefice_merch` are **not** sent by webhook. They default to `0` and are filled in by the employee in the dashboard.

### Vendor auto-creation

If `vendor_id` from the webhook doesn't exist in the `vendors` table:
- A new vendor is automatically created using `vendor_name` from the meta
- The vendor can then be edited (phone added etc.) in the Vendors page

### Signature validation

Every request is validated using HMAC-SHA256:
```
signature = HMAC-SHA256(raw_body, WEBHOOK_SECRET)
```
Uses the `x-wc-webhook-signature` header sent by WooCommerce.

### Upsert behavior

Orders are upserted on `order_ref`. If the same order number fires again (e.g. status update from WooCommerce), it updates rather than duplicating.

- Single admin account via Supabase Auth (email + password)
- All dashboard routes protected by `src/middleware.ts`
- Unauthenticated users are redirected to `/login`

## 10. Key Business Rules

1. **An order can only be in one payout.** Once linked via `payout_orders`, it no longer appears in the eligible orders list.

2. **Payout net = LIVREE benefice minus RETOUR production cost.** Retours are charged back to the vendor at `prix_vendeur` (not `prix_fournisseur`).

3. **Vendor never sees our real cost.** `prix_fournisseur` and `benefice_merch` are internal only. Vendors only see `prix_vendeur`.

4. **`prix_fournisseur` and `benefice_merch` must be entered manually.** The webhook only brings `prix_client`. Everything else is filled by an employee.

5. **Order status is independent of payment status.** An order can be LIVREE but `is_vendor_paid = false` until a payout is created.

6. **Vendor auto-creation.** If WooCommerce sends a `vendor_id` that doesn't exist in the DB, the vendor is created automatically using `vendor_name` from the meta.

## 11. DB Migration (Pricing Column Rename)

If migrating from the original schema (which used `product_cost`, `printing_cost`, `merch_benefice`), run `migration_pricing.sql` in the Supabase SQL Editor.

This migration:
1. Adds the new columns (`prix_fournisseur`, `benefice_merch`, `prix_client`, etc.)
2. Copies existing data from old columns
3. Drops and recreates computed columns with correct formulas
4. Recreates all 3 SQL views with updated column names
